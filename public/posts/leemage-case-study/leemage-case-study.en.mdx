## How Leemage Grew from a Cost Problem into a File Management Platform

When building side projects, file uploads usually get pushed to the end.

You start with authentication, deployment, payments, and the core product flow, and only later think, "I should probably support image uploads too." I used to treat uploads as a minor feature as well. But after repeating the same pattern across multiple projects, my perspective changed. The moment image uploads entered the picture, cost became a very practical issue.

The feature itself looks simple, but depending on the service you choose, fixed costs show up sooner than expected. That makes the problem awkward for side projects. Building it yourself feels tedious, but paying for an external SaaS every time also feels wasteful. Eventually I ended up asking a simple question:

> Do I really have to solve this problem with a paid service every single time?

That was when OCI Object Storage and Cloudflare R2 became interesting options. Both provide meaningful free-tier capacity, and at side-project scale they felt realistic enough to build around. So instead of choosing a fully managed service like Cloudinary, I decided to move toward a setup where I could control cost more directly. That decision led to `Leemage`.

`Leemage` is a self-hosted file management platform designed around cost control. It organizes files by project, converts heavy PNG assets into more efficient formats such as WebP during the upload flow, and exposes both a REST API and an SDK so the same system can be reused across multiple projects instead of staying trapped inside one app.

## It Started as an Upload Feature, but Turned into an Operations Problem

The original goal was straightforward: reduce upload costs while keeping the developer experience practical enough to use in a real project.

Once I started building it, I realized the real problem was not simply storing files. The more painful part was everything that came after storage: management, repeated reuse, and keeping things organized across projects. As more projects accumulate, the important question becomes less about "Can I upload this file?" and more about "Where is it stored, what variants exist, and can I reuse it later?"

That is why `Leemage` did not stop at a single upload endpoint. I made files separable by project, used presigned URL uploads to keep the API server lightweight, and added an explicit confirm step so image transformation and variant management could happen after the upload completed. In other words, the architecture spent more time thinking about what happens after storage than before it.

## Leemage Architecture

The overall flow is fairly simple.

1. The client first requests a presigned upload URL.
2. The API creates a pending file record and returns the presigned URL.
3. The client uploads the file directly to object storage.
4. During the confirm step, `Leemage` validates the uploaded object and collects metadata.
5. If the file is an image, `Sharp` generates optimized variants such as WebP and stores them back in object storage.
6. Finally, the database stores the file metadata and variant references so other projects can consume them through the API or SDK.

What mattered to me here was not the upload itself, but the separation of concerns in the flow. I did not want a heavy upload server, and I did not want storage to be tightly coupled to one provider. If you want to balance cost, performance, and operational convenience, it helps to think of the upload path, storage path, and delivery path as separate layers.

Another design choice I cared about was treating `OpenAPI` as a single source of truth instead of just a documentation artifact. In `Leemage`, the web API docs UI, the YAML exposed at `/api/v1/openapi`, and the SDK synchronization flow are all connected around the same OpenAPI spec. I think that matters more in the long run than it first appears, because it prevents the "human-facing docs" and the "actual contract" from slowly drifting apart.

## I Started with Cost, but Ended Up Caring about Performance Too

One of the more interesting parts of building `Leemage` was how quickly cost concerns turned into performance concerns as well. Image conversion is the clearest example. Instead of storing and serving a heavy PNG as-is, converting it to WebP reduces both storage usage and transfer size. That is not just about making an image lighter. It directly affects storage cost, bandwidth cost, and the speed users experience.

![10GB Cost Comparison](/public/posts/leemage-case-study/leemage-cost-comparison.png)

If you imagine storing around 10GB of files, the difference between a typical paid storage setup and a free-tier-centered structure becomes obvious. That gap was the original motivation behind `Leemage`. In side projects, even when traffic is not huge, repeated uploads and a growing image library start creating cost pressure sooner than expected. I wanted a structure where that cost could be managed instead of merely accepted.

![PNG vs WebP Reduction](/public/posts/leemage-case-study/leemage-webp-reduction.png)

The size reduction from converting PNG to WebP was also meaningful in practice. The [Google WebP documentation](https://developers.google.com/speed/webp) explains that lossless WebP images can be about 26% smaller than PNG. The exact number will always depend on the image itself and the encoding setup, but the pattern matters more than the exact percentage. Once you start thinking of upload as the first step of optimization rather than simple storage, adding CDN delivery and caching strategies becomes much more natural.

## The Moment It Started Feeling Like a Reusable Foundation

`Leemage` started to feel less like an internal tool and more like an actual product once I reused it in other projects.

Reusing a previously built upload flow makes a bigger difference than it sounds. It means a new project does not have to redesign uploads from scratch, and it can start with storage paths, URL rules, file lifecycle management, and deletion behavior already defined.

In practice, I also used `Leemage` as a storage adapter in another side project called [`leesfield`](https://github.com/leey00nsu/leesfield). That experience changed how I saw the project. Before that, it was closer to "a tool I made because I was personally annoyed." After that, it started feeling more like infrastructure that reduces repeated work. The same upload feature becomes much more valuable when it shortens the startup cost of the next project.

The moment you open an API to outside consumers, documentation quality starts to resemble product quality. That is why I did not want OpenAPI to be treated as a separate documentation chore. Keeping the spec, the web docs, and the SDK sync flow aligned was one of the most important decisions in `Leemage`.

## Closing Thoughts

Looking back, the most important part of `Leemage` was not a flashy technical choice. It was the decision to recognize file handling as a repeated problem and to avoid solving it in a one-off way.

Even in side projects, I try to focus less on whether I can implement a feature quickly and more on whether that feature represents a recurring problem. If it does, I want to leave behind something reusable. `Leemage` is probably the clearest example of that tendency in my work. It started from cost pressure, but it eventually became a project about operability, extensibility, and reuse.
