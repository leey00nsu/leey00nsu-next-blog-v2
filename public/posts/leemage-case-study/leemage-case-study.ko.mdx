## 비용 문제에서 시작해 파일 관리 플랫폼이 된 Leemage 제작기

사이드 프로젝트를 만들다 보면 파일 업로드는 늘 뒤로 밀립니다.

로그인, 배포, 결제, 제품에서 먼저 보여야 할 기능을 만들고 나면 그제서야 "이미지도 올려야지" 하는 식으로 붙게 됩니다. 저도 처음에는 파일 업로드를 그냥 부가 기능 정도로 생각했습니다. 그런데 프로젝트를 몇 번 반복하다 보니 생각이 바뀌었습니다. 특히 이미지 업로드가 들어가는 순간부터 비용이 꽤 현실적인 문제가 됐습니다.

기능 자체는 작아 보이는데, 어떤 서비스를 선택하느냐에 따라 생각보다 빨리 고정비가 생깁니다. 사이드 프로젝트 입장에서는 이 지점이 꽤 애매합니다. 직접 만들기에는 번거롭고, 그렇다고 매번 외부 SaaS 비용을 감수하기에는 아깝습니다. 그때 이런 질문이 남았습니다.

> 정말 이 문제를 계속 유료 서비스로만 해결해야 할까?

그때 눈에 들어온 것이 OCI Object Storage와 Cloudflare R2였습니다. 두 서비스 모두 Free Tier에서 일정 수준의 용량을 무료로 제공했고, 사이드 프로젝트 규모에서는 충분히 현실적인 선택지라고 판단했습니다. 그래서 Cloudinary 같은 완성형 서비스 대신 비용을 더 직접 통제할 수 있는 방향으로 가보기로 했습니다. 그렇게 시작한 것이 `Leemage`입니다.

`Leemage`는 비용 절감을 위한 자체 호스팅 파일 관리 플랫폼입니다. 프로젝트 단위로 파일을 정리할 수 있고, 업로드 과정에서 무거운 PNG를 WebP 같은 더 효율적인 포맷으로 변환할 수 있습니다. 여기에 REST API와 SDK를 붙여 다른 프로젝트에도 빠르게 연결할 수 있게 만들었습니다.

## 처음에는 "업로드 기능"이었지만 점점 "운영 문제"가 됐다

처음 목표는 확실했습니다. 업로드 비용을 줄이면서도 실제 프로젝트에 바로 붙일 수 있는 수준의 경험을 만드는 것이었습니다.

그런데 막상 만들기 시작하니 제가 불편했던 건 파일을 저장하는 순간보다 그 이후였습니다. 프로젝트가 늘어날수록 파일은 계속 쌓였고, 나중에는 "어디에 있고, 어떤 버전이 있고, 다시 가져다 쓸 수 있는가"를 찾는 시간이 더 거슬렸습니다.

그래서 `Leemage`는 업로드 엔드포인트 하나로 끝나지 않았습니다. 프로젝트별로 파일을 구분할 수 있게 했고, Presigned URL 기반 업로드로 서버 부하를 줄였고, 업로드 완료 후에는 이미지 변환과 variant 관리까지 이어지도록 만들었습니다. 저장 이전보다 저장 이후의 일이 더 많았습니다.

## Leemage 아키텍처

전체 동작은 복잡하지 않습니다.

1. 클라이언트가 먼저 presign 요청을 보냅니다.
2. API는 pending 상태의 파일 레코드를 만들고 presigned URL을 반환합니다.
3. 클라이언트는 파일을 스토리지에 직접 업로드합니다.
4. 이후 confirm 단계에서 `Leemage`가 업로드된 파일을 기준으로 메타데이터를 정리합니다.
5. 이미지라면 `Sharp`를 이용해 WebP 같은 variant를 생성하고 다시 저장합니다.
6. 최종적으로 파일 메타데이터와 variant 정보가 DB에 정리되고, 이후에는 API나 SDK를 통해 다른 프로젝트에서도 재사용할 수 있습니다.

이 구조에서 제가 보고 있던 건 업로드 버튼 하나가 아니라 업로드 이후의 경로였습니다. 업로드 서버를 무겁게 만들고 싶지 않았고, 저장소 의존성을 하나의 서비스에 묶고 싶지도 않았습니다. 비용, 성능, 운영 편의성을 같이 보려면 업로드 경로와 저장 경로, 전달 경로를 나눠두는 편이 낫다고 판단했습니다.

개인적으로 신경 쓴 부분은 `OpenAPI`였습니다. `Leemage`에서는 API 문서 웹 UI, `/api/v1/openapi`로 제공되는 YAML, SDK 동기화가 하나의 OpenAPI 스펙을 기준으로 움직입니다. 외부 프로젝트에서 붙일 때 "문서용 설명"과 "실제 스펙"이 어긋나기 시작하면 나중에 더 귀찮아질 것 같았습니다.

## 비용을 줄이기 위해 만들었지만 성능까지 같이 보게 됐다

`Leemage`를 만들면서 흥미로웠던 건 비용 문제를 파고들수록 자연스럽게 성능 문제와 맞닿게 된다는 점이었습니다. 대표적인 예가 이미지 변환입니다. 무거운 PNG를 그대로 저장하고 전달하는 대신 WebP로 변환하면 저장 용량과 전송량을 동시에 줄일 수 있습니다. 이건 단지 "이미지가 가벼워진다"는 수준의 이야기가 아니라, 스토리지 비용과 대역폭 비용, 그리고 사용자 체감 속도까지 한 번에 연결되는 문제였습니다.

![10GB Cost Comparison](/public/posts/leemage-case-study/leemage-cost-comparison.png)

예를 들어 10GB 정도의 파일을 저장한다고 가정하면, 일반적인 유료 스토리지 사용과 Free Tier 기반 구조 사이에는 차이가 꽤 벌어집니다. 제가 `Leemage`를 만든 출발점도 이 지점이었습니다. 사이드 프로젝트에서는 트래픽이 아주 크지 않아도 반복되는 업로드와 이미지 자산 누적으로 인해 비용 압박이 생각보다 빨리 시작됩니다. 그 비용을 그냥 받아들이기보다 직접 통제 가능한 구조로 가져오고 싶었습니다.

![PNG vs WebP Reduction](/public/posts/leemage-case-study/leemage-webp-reduction.png)

PNG 대비 WebP 변환으로 파일 크기를 줄일 수 있다는 점도 체감이 컸습니다. [Google WebP 문서](https://developers.google.com/speed/webp?hl=ko)에서는 무손실 WebP 이미지가 PNG보다 약 26% 더 작다고 설명합니다. 실제 결과는 이미지 특성이나 인코딩 방식에 따라 달라지겠지만, 저는 그 경향만으로도 충분히 볼 만하다고 생각했습니다. 업로드를 저장으로 끝내지 않고 최적화의 시작점으로 보면, CDN 적용이나 캐싱 전략도 이어서 생각하기 쉬워집니다.

## 내부 도구가 아니라 재사용 가능한 기반이 되기 시작한 시점

`Leemage`가 작은 내부 도구에서 조금 더 제품처럼 보이기 시작한 건 다른 프로젝트에 실제로 재사용하기 시작하면서부터였습니다.

한 번 만들어둔 업로드 흐름을 다시 붙일 수 있다는 건 생각보다 큰 차이를 만듭니다. 새로운 프로젝트를 시작할 때 파일 업로드를 처음부터 다시 설계하지 않아도 되고, 스토리지와 URL 구조, 파일 관리 방식, 삭제 전략까지 이미 정리된 상태에서 출발할 수 있기 때문입니다.

실제로 `Leemage`는 다른 사이드 프로젝트인 [`leesfield`](https://github.com/leey00nsu/leesfield)에서도 저장 어댑터로 사용했습니다. 이때 체감이 컸습니다. 새 프로젝트에서 다시 업로드 방식을 고르고, URL 규칙을 정하고, 삭제 전략을 만들지 않아도 됐습니다. 같은 업로드 기능이라도 재사용 가능한 구조로 남겨두면 다음 프로젝트를 시작할 때 생각할 것이 줄어듭니다.

API를 외부에 열어두면 문서와 실제 동작이 조금만 어긋나도 바로 불편해집니다. 그래서 OpenAPI를 별도 문서 작업으로 취급하지 않고, 실제 스펙을 기준으로 웹 문서와 배포 산출물, SDK 동기화까지 이어지게 했습니다.

## 마무리

`Leemage`에서 남은 건 화려한 기술 선택보다 반복되는 문제를 어떻게 남겨둘지에 대한 감각이었습니다. 한 번 해결하고 끝낼 문제인지, 다음 프로젝트에서도 다시 나올 문제인지가 생각보다 자주 떠올랐습니다.

사이드 프로젝트에서도 자주 다시 만날 문제라면 가능한 한 재사용 가능한 형태로 남기고 싶습니다. `Leemage`는 그런 성향이 가장 잘 드러난 프로젝트였습니다. 시작은 비용이었지만, 만들다 보니 운영, 확장, 재사용까지 같이 챙기게 됐습니다.
