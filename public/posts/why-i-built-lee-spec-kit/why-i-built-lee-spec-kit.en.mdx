## Why I Started Building Work Structure Before Asking AI to Code

When people talk about building software with AI agents, the phrase [`harness engineering`](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html) comes up more and more often. At first it sounded a little grandiose to me. But after repeatedly working with AI coding assistants, it started to feel like a very precise description.

When I work with AI coding assistants, the same model can feel sharp in one repo and lost in another. The difference is often not the model. It is whether the repo tells the assistant where the docs are, what rules to follow, what stage the work is in, and what should happen next. That is why the word `harness` started to make sense to me.

![AI Harness Structure](/public/posts/why-i-built-lee-spec-kit/ai-harness-structure.png)

That thought came up again while reading about [`spec-driven development`](https://specdriven.ai/). Defining the problem before implementation, recording decisions, and leaving the work status visible are useful when people work together. With AI, they show up in daily work more quickly. A person can sometimes fill in missing context from experience. An assistant often treats that missing context as room to drift.

`lee-spec-kit` started from that problem. I built it as a CLI to create the docs skeleton quickly and keep feature work moving through `spec`, `plan`, `task`, `decision`, and `PR` in the same shape each time. Instead of rewriting a better prompt for every session, I wanted a structure the assistant could keep referring to.

## The Part That Slowed Me Down Was Context Transfer

In practice, code generation was not always the slow part. The slower part was telling the assistant the same context again.

This was not a problem I discovered only after I started working with AI. I had seen the same pattern many times while working at a company. The code was there, but the intent behind it was often missing. You could see the final result, but not the planning context or the decision-making process that led to it. In practice, a new developer often needed help from someone who already knew the background in order to truly understand the codebase.

Every project has different docs, different rules, and different ways of expressing progress. When that happens, you end up repeating the same explanations to AI over and over again. Where was this feature discussed? What decisions led here? Are we in the spec stage or the plan stage? What still needs to be checked before a PR? If that information is not organized consistently, humans get confused and AI gets unstable.

Under those conditions, even a very good model struggles to produce stable output. Every new repository effectively begins with the question, "How does this team work?" I did not want to patch that problem with a few extra prompt lines. I wanted the documentation structure and workflow to carry more of that context from the beginning. Reducing repeated explanation felt like a more direct way to lower the cost of AI collaboration.

## So lee-spec-kit Starts by Shaping the Documentation Flow

That is why `lee-spec-kit` is designed to quickly bootstrap the basic documentation skeleton of a project and preserve a feature-level flow through `spec`, `plan`, `task`, `decision`, and `PR` stages. I cared less about adding many templates and more about putting context in predictable places, so humans and AI could talk about project state with the same words.

What was decided, why it was decided, how far the work has progressed, and what should happen next should all remain visible in the document structure itself.

![Feature Workflow Pipeline](/public/posts/why-i-built-lee-spec-kit/feature-workflow-pipeline.png)

This idea did not appear out of nowhere. I was influenced by the structure of `spec-kit` and `OpenSpec`. Both projects were compelling because they organize work around specifications and preserve the thinking process before implementation. Seeing those projects made me think this approach would work well with AI collaboration too.

## But I Wanted It to Fit My Workflow

At the same time, I wanted to shrink that idea into something that fit the way I actually work. I needed a CLI that could be applied lightly and quickly. The feature-level flow still had to be visible, and the documentation structure had to stay lightweight without becoming vague for AI.

At this point, a natural question follows: "Why not use an existing tool?" Someone could reasonably think of options like `oh my opencode`. That is a fair question. But I wanted to design a harness for my own working context, not adopt another general-purpose AI development tool and stop there.

I did not want to remain only a user who knows how to use good tools. I wanted to work directly on the question of how to structure my own work system. `lee-spec-kit` came out of that choice.

![Tailored Workflow System](/public/posts/why-i-built-lee-spec-kit/tailored-workflow-system.png)

## Closing Thoughts

Building this changed how I start new features. I now try to write down what I am making, why that direction was chosen, where the work is, and what should happen next before asking an assistant to continue.

I do not see `lee-spec-kit` as a grand methodology. It is a small workbench for reducing repeated explanation when working with AI. A strong model helps, but without records and order it still loses the thread quickly.

`lee-spec-kit` is the tool I made to reduce that problem in my own workflow.
